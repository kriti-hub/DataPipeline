# Security and Compliance

> WellNow Staffing Analytics -- Healthcare People Analytics Pipeline POC

This document describes the security posture, privacy safeguards, and compliance
considerations for the WellNow Staffing Analytics data pipeline. Although this
proof-of-concept operates exclusively on synthetic data, every design decision
anticipates the controls that a production deployment handling real PHI/PII
would require.

---

## 1. Synthetic Data Declaration

| Property | Detail |
|---|---|
| **Data source** | 100% synthetic, generated by the Python `Faker` library |
| **Real patient data** | None -- no real patients appear anywhere in the pipeline |
| **Real employee PII** | None -- no real names, emails, phone numbers, or addresses |
| **Protected Health Information (PHI)** | None -- no diagnoses, treatments, or medical record numbers |
| **Identifiers** | All employee IDs, email addresses, and names are fictional |
| **Statistical distributions** | Data patterns (turnover rates, overtime hours, headcount curves) mimic realistic healthcare staffing distributions but are not derived from actual records |

Because no real data is present, this POC is not subject to HIPAA enforcement.
Nevertheless, every pipeline component is designed as though real PHI/PII could
flow through it, so that upgrading to production requires configuration changes
rather than architectural changes.

## 2. Authentication Patterns

### 2.1 Simulated HRIS API

| Mechanism | Implementation |
|---|---|
| **Method** | `X-API-Key` header on every HTTP request |
| **Enforcement** | Middleware validates the key before any endpoint handler executes |
| **Excluded endpoint** | `GET /health` -- unauthenticated so load balancers and uptime monitors can probe without credentials |
| **Key storage** | Environment variable `HRIS_API_KEY` (see Secret Management below) |
| **Failure response** | `401 Unauthorized` with no key detail leaked in the response body |

### 2.2 BigQuery

| Mechanism | Implementation |
|---|---|
| **Method** | IAM-based access via GCP service account |
| **Credential type** | Service account JSON key (local dev) or Workload Identity (Cloud Run / Cloud Functions) |
| **Least privilege** | Service account granted only `bigquery.dataEditor` and `bigquery.jobUser` on the target dataset |

### 2.3 Dashboard

| Mechanism | Implementation |
|---|---|
| **POC state** | Public access -- no authentication layer |
| **Production pattern** | Identity-Aware Proxy (IAP) or SSO integration (SAML/OIDC) with role-based access to dashboard views |

## 3. Secret Management

### 3.1 Development Environment

- Secrets are stored in a `.env` file at the project root.
- `.env` is excluded from version control via `.gitignore` (see Section 8).
- `.env.example` provides a safe template with placeholder values and
  documentation comments -- it contains **no actual credentials**.

### 3.2 Production Pattern

- **GCP Secret Manager** is the target secret store.
- `infrastructure/scripts/setup_secrets.sh` documents the full setup procedure
  for creating and versioning secrets in Secret Manager.
- Cloud Run and Cloud Functions reference secrets by name at deploy time;
  values are injected as environment variables at runtime.

### 3.3 Verification

| Check | How |
|---|---|
| No credentials in source | CI lint step scans for high-entropy strings and common secret patterns |
| `.gitignore` coverage | Reviewed to ensure `.env`, `*.key`, `*.pem`, `service-account*.json`, and `credentials*.json` are all excluded |
| Template safety | `.env.example` is audited to confirm every value is a placeholder (`your-*-here`) |

## 4. Audit Trail Design

The pipeline maintains three utility tables in BigQuery that together provide a
complete, queryable audit trail of every ETL run.

### 4.1 `_pipeline_runs`

Tracks every ETL execution from start to finish.

| Column | Purpose |
|---|---|
| `run_id` | UUID for the execution |
| `started_at` | Timestamp when the run began |
| `completed_at` | Timestamp when the run finished (NULL if still running) |
| `status` | `running`, `success`, or `failed` |
| `records_extracted` | Count of rows pulled from the API |
| `records_loaded` | Count of rows written to BigQuery |
| `records_quarantined` | Count of rows that failed data quality checks |
| `error_message` | Captured exception text on failure |
| `_batch_id` | Cross-reference key shared with related tables |

### 4.2 `_data_quality_log`

Records the outcome of every data quality check applied during a run.

| Column | Purpose |
|---|---|
| `log_id` | Unique identifier for the check |
| `run_id` | Links to `_pipeline_runs` |
| `rule_name` | Human-readable name of the DQ rule (e.g., `not_null_employee_id`) |
| `check_result` | `pass` or `fail` |
| `records_checked` | Total rows evaluated |
| `records_passed` | Rows that passed |
| `records_failed` | Rows that failed |
| `checked_at` | Timestamp of the check |
| `_batch_id` | Cross-reference key |

### 4.3 `_quarantine`

Preserves every rejected record with full context for investigation and
reprocessing.

| Column | Purpose |
|---|---|
| `quarantine_id` | Unique identifier |
| `run_id` | Links to `_pipeline_runs` |
| `source_record` | JSON-serialized original record |
| `failure_reason` | Which DQ rule(s) the record violated |
| `quarantined_at` | Timestamp |
| `_batch_id` | Cross-reference key |

### 4.4 Cloud Logging

- **API access logs**: Every request to the HRIS API (method, path, status code,
  latency, caller IP) is captured by Cloud Run's built-in request logging.
- **ETL execution logs**: Structured logs emitted by the pipeline are shipped to
  Cloud Logging with severity levels (`INFO`, `WARNING`, `ERROR`).
- **Log retention**: Default 30-day retention in Cloud Logging; production
  deployments should export to a long-term BigQuery log sink for compliance
  retention requirements.

## 5. HIPAA Considerations

Even though this POC contains no real PHI, the architecture is designed with
HIPAA's core requirements in mind. The table below maps each HIPAA safeguard to
the corresponding pipeline control.

| HIPAA Safeguard | Pipeline Implementation |
|---|---|
| **Minimum Necessary Rule** | The API returns only the fields required for workforce analytics. No SSN, date of birth, or medical records are included in any API response or BigQuery table. |
| **Access Control** | API key authentication on the HRIS API; IAM-based access control on BigQuery datasets and tables. |
| **Audit Logging** | Complete execution trail via `_pipeline_runs`, `_data_quality_log`, and `_quarantine` tables. Cloud Logging captures API access and ETL events. |
| **Encryption at Rest** | BigQuery and Google Cloud Storage encrypt all data at rest by default using Google-managed encryption keys (AES-256). |
| **Encryption in Transit** | HTTPS is enforced on Cloud Run. All API calls between the ETL pipeline and the HRIS API use TLS. BigQuery client libraries use TLS for all data transfers. |
| **Data Retention** | The fact table is partitioned by date, enabling time-based lifecycle policies for automated expiration and deletion of aged data. |
| **Business Associate Agreement (BAA)** | Required for any production deployment handling real PHI. GCP offers a BAA for healthcare customers covering BigQuery, Cloud Storage, Cloud Run, and other services used in this architecture. |

## 6. What Changes for Production with Real PHI/PII

Moving from synthetic data to real Protected Health Information or Personally
Identifiable Information requires the following additional controls.

### 6.1 Encryption Enhancements

- **Customer-Managed Encryption Keys (CMEK)** in BigQuery and Cloud Storage,
  managed through Cloud KMS, so the organization retains full control over key
  rotation and revocation.
- **Field-level encryption or tokenization** for PHI/PII columns (e.g., names,
  employee IDs) so that even authorized BigQuery users see only tokens unless
  explicitly granted decryption access.

### 6.2 Network and Perimeter Controls

- **VPC Service Controls** to create a security perimeter around BigQuery and
  Cloud Storage, preventing data exfiltration to unauthorized projects.
- **Cloud Armor** for API protection -- rate limiting, IP allowlisting, and
  WAF rules on the Cloud Run service.
- **Private Google Access** so that ETL workloads communicate with BigQuery
  over Google's internal network rather than the public internet.

### 6.3 Access Control Hardening

- **Real IAM with least-privilege service accounts** -- separate service
  accounts for the API, ETL pipeline, and dashboard, each with the minimum
  roles required.
- **BigQuery row-level security** and column-level security policies for
  role-based access to sensitive fields.
- **RBAC on the dashboard** with SSO integration (SAML 2.0 or OIDC) so that
  only authorized roles can view specific metrics.

### 6.4 Data Protection and Governance

- **Data Loss Prevention (DLP) API** scanning on BigQuery tables and Cloud
  Storage buckets to detect and flag any unexpected PHI/PII.
- **Data retention and deletion policies** aligned with HIPAA requirements,
  enforced via BigQuery table expiration and Cloud Storage lifecycle rules.
- **Right-to-delete workflows** for employee data subject requests.

### 6.5 Operational Security

- **Regular security audits and penetration testing** on the API and
  infrastructure.
- **Business Associate Agreement (BAA)** executed with GCP before any real
  PHI enters the pipeline.
- **Incident response plan** documented and tested for data breach scenarios.

## 7. Additional Compliance Awareness

The pipeline's design supports readiness for several compliance frameworks
beyond HIPAA.

### 7.1 SOC 2 Type II

| Principle | Pipeline Support |
|---|---|
| **Security** | API authentication, IAM, encryption at rest and in transit |
| **Availability** | Cloud Run auto-scaling, Cloud Scheduler for reliable execution |
| **Processing Integrity** | Data quality checks, quarantine table, audit trail |
| **Confidentiality** | Secret management, `.gitignore` controls, least-privilege IAM |
| **Change Management** | All infrastructure and pipeline code version-controlled in Git with PR-based review |

### 7.2 CCPA / State Privacy Laws

- **Data minimization**: The API and schema include only the fields necessary
  for workforce analytics -- no unnecessary PII is collected or stored.
- **Transparency**: The data dictionary (`docs/data_dictionary.md`) documents
  every field and its purpose.
- **Deletion readiness**: Partitioned tables and the quarantine design support
  targeted record deletion.

### 7.3 EEOC / Pay Equity

- The dimensional schema includes fields for `department`, `position_title`,
  `employment_type`, and compensation metrics.
- This structure supports future demographic analysis for pay equity audits
  and EEOC reporting without requiring schema changes.

### 7.4 OSHA / Labor Law Compliance

- Overtime hours and workload metrics are captured in the fact table.
- Dashboard KPIs surface overtime trends and staffing ratios that support
  compliance monitoring for labor law requirements.
- Alerting thresholds can flag when overtime patterns approach regulatory
  limits.

## 8. `.gitignore` Security Items

The project `.gitignore` excludes the following categories of sensitive files
from version control.

| Pattern | Category | Reason |
|---|---|---|
| `.env` | Secrets | Contains API keys, project IDs, and configuration values |
| `.env.local` | Secrets | Local override file that may contain developer-specific credentials |
| `.env.production` | Secrets | Production credentials -- must never appear in source control |
| `*.key` | Certificates | Private keys for TLS or signing |
| `*.pem` | Certificates | PEM-encoded certificates and private keys |
| `service-account*.json` | GCP credentials | Service account key files used for local development |
| `credentials*.json` | GCP credentials | OAuth or service account credential files |
| `*.csv` | Data files | Raw or processed data that may contain sensitive records |
| `data/raw/` | Data files | Landing zone for extracted data -- may contain unvalidated records |
| `data/staging/` | Data files | Intermediate transformed data awaiting load |

Additional exclusions (IDE configs, Python/Node caches, OS files, build
artifacts) are documented in the `.gitignore` file itself but do not carry
security implications.
